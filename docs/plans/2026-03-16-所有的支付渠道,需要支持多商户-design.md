## 1. 统一语言术语表

| 术语 | 含义 | 类型 |
|------|------|------|
| **Merchant**（商户） | 使用支付平台的业务主体，持有各渠道的支付账户凭据 | 聚合根 |
| **MerchantID**（商户ID） | 商户的全局唯一标识符 | 值对象 |
| **MerchantStatus**（商户状态） | 商户的激活状态（ACTIVE / SUSPENDED / DEACTIVATED） | 值对象（枚举） |
| **ChannelCredential**（渠道凭据） | 商户在某一支付渠道（CARD/PAYPAL）的密钥配置，如 API Key、ClientID+Secret | 实体 |
| **ChannelCredentialID**（凭据ID） | 渠道凭据的唯一标识 | 值对象 |
| **PaymentChannel**（支付渠道） | 支付方式枚举，与 payment 上下文的 `PaymentMethod` 对齐（CARD/PAYPAL） | 值对象（枚举） |
| **CredentialStatus**（凭据状态） | 某渠道凭据的启用状态（ACTIVE / REVOKED） | 值对象（枚举） |
| **MerchantRegistered**（商户已注册） | 商户完成注册后触发 | 领域事件 |
| **CredentialAdded**（凭据已添加） | 商户为某渠道添加凭据后触发 | 领域事件 |
| **CredentialRevoked**（凭据已吊销） | 商户某渠道凭据被吊销后触发 | 领域事件 |
| **MerchantSuspended**（商户已暂停） | 商户被暂停时触发 | 领域事件 |
| **MerchantQuery**（商户查询端口） | payment 上下文通过此端口查询有效渠道凭据，ACL 边界 | 端口（Query） |
| **ChannelCredentialView**（凭据视图） | payment 视角的凭据 DTO，只含网关初始化所需的字段 | 值对象（DTO） |

## 2. 上下文归属

### 结论：新建 `merchant` 上下文

**理由：**
- **单一职责**：商户管理（注册、凭据生命周期、状态）是独立业务能力，与支付交易流程的职责正交。
- **变更频率不同**：商户凭据很少变动（配置类），交易高频发生（操作类），混入 `payment` 会污染其聚合。
- **潜在扩展**：未来可能有商户结算、费率配置、风控规则等，独立上下文承载能力更强。

### 跨上下文交互

```
merchant 上下文                    payment 上下文
──────────────────                 ──────────────────────────────────
Merchant Aggregate                 ChargeUseCase
ChannelCredential Entity  ←ACL→   port.MerchantQuery (payment 侧定义)
MerchantRepository                 adapter/merchant/merchant_adapter.go
                                   (实现 MerchantQuery，跨上下文 import 仅在 adapter/)
```

**映射策略：防腐层（ACL）**
- `payment` 上下文在 `domain/port/merchant.go` 定义 `MerchantQuery` 接口（消费方定义端口原则）
- `payment/adapter/merchant/merchant_adapter.go` 跨上下文 import `merchant` 的 `port.MerchantRepository`，翻译为 `payment` 侧的 `ChannelCredentialView`
- `payment` 内部网关调用时，由 `ChargeUseCase` 通过 `MerchantQuery` 获取凭据，动态构造对应渠道的 Gateway 实例（或从注册表中按 MerchantID+Channel 路由）

## 4. 聚合根设计

### `merchant/domain/model/merchant.go`

```go
package model

import (
    "time"
    "github.com/google/uuid"
    "payment-demo/internal/merchant/domain/event"
)

// ── 标识符 ───────────────────────────────────────────────
type MerchantID string

func NewMerchantID() MerchantID { return MerchantID(uuid.New().String()) }

// ── 状态枚举 ──────────────────────────────────────────────
type MerchantStatus string

const (
    MerchantStatusActive      MerchantStatus = "ACTIVE"
    MerchantStatusSuspended   MerchantStatus = "SUSPENDED"
    MerchantStatusDeactivated MerchantStatus = "DEACTIVATED"
)

// ── 渠道枚举（对齐 payment.PaymentMethod） ────────────────
type PaymentChannel string

const (
    ChannelCard   PaymentChannel = "CARD"
    ChannelPayPal PaymentChannel = "PAYPAL"
)

// ── ChannelCredential 实体 ────────────────────────────────
type ChannelCredentialID string

func NewChannelCredentialID() ChannelCredentialID {
    return ChannelCredentialID(uuid.New().String())
}

type CredentialStatus string

const (
    CredentialStatusActive  CredentialStatus = "ACTIVE"
    CredentialStatusRevoked CredentialStatus = "REVOKED"
)

// ChannelCredential 商户在某支付渠道的接入凭据（实体）
// Secrets 存储渠道密钥，key 为字段名（如 "api_key"、"client_id"、"client_secret"）
// 不在聚合根层做序列化，Infra 层负责加密存储
type ChannelCredential struct {
    ID         ChannelCredentialID
    MerchantID MerchantID
    Channel    PaymentChannel
    Secrets    map[string]string // {"api_key": "sk_live_xxx"} / {"client_id":"...", "client_secret":"..."}
    Status     CredentialStatus
    CreatedAt  time.Time
    RevokedAt  *time.Time
}

// ── Merchant 聚合根 ───────────────────────────────────────
// 不变式：同一渠道最多只有一个 ACTIVE 凭据
type Merchant struct {
    ID          MerchantID
    Name        string
    Status      MerchantStatus
    Credentials []ChannelCredential
    CreatedAt   time.Time
    Events      []event.DomainEvent
}

// NewMerchant 工厂方法，初始状态 ACTIVE，触发 MerchantRegistered 事件
func NewMerchant(name string) *Merchant {
    m := &Merchant{
        ID:        NewMerchantID(),
        Name:      name,
        Status:    MerchantStatusActive,
        CreatedAt: time.Now(),
    }
    m.addEvent(event.MerchantRegistered{
        MerchantID: string(m.ID),
        Name:       name,
        OccurredAt: m.CreatedAt,
    })
    return m
}

// AddCredential 为商户添加渠道凭据
// 不变式：若该渠道已有 ACTIVE 凭据，须先吊销旧凭据（防止重复激活）
func (m *Merchant) AddCredential(channel PaymentChannel, secrets map[string]string) error {
    if m.Status != MerchantStatusActive {
        return ErrMerchantNotActive
    }
    for _, c := range m.Credentials {
        if c.Channel == channel && c.Status == CredentialStatusActive {
            return ErrCredentialAlreadyExists
        }
    }
    cred := ChannelCredential{
        ID:         NewChannelCredentialID(),
        MerchantID: m.ID,
        Channel:    channel,
        Secrets:    secrets,
        Status:     CredentialStatusActive,
        CreatedAt:  time.Now(),
    }
    m.Credentials = append(m.Credentials, cred)
    m.addEvent(event.CredentialAdded{
        MerchantID:   string(m.ID),
        CredentialID: string(cred.ID),
        Channel:      string(channel),
        OccurredAt:   cred.CreatedAt,
    })
    return nil
}

// RevokeCredential 吊销指定渠道凭据
// ACTIVE → REVOKED，非法状态转换返回 ErrInvalidStateTransition
func (m *Merchant) RevokeCredential(credentialID ChannelCredentialID) error {
    for i, c := range m.Credentials {
        if c.ID == credentialID {
            if c.Status != CredentialStatusActive {
                return ErrInvalidStateTransition
            }
            now := time.Now()
            m.Credentials[i].Status = CredentialStatusRevoked
            m.Credentials[i].RevokedAt = &now
            m.addEvent(event.CredentialRevoked{
                MerchantID:   string(m.ID),
                CredentialID: string(credentialID),
                Channel:      string(c.Channel),
                OccurredAt:   now,
            })
            return nil
        }
    }
    return ErrCredentialNotFound
}

// Suspend 暂停商户：ACTIVE → SUSPENDED
func (m *Merchant) Suspend() error {
    if m.Status != MerchantStatusActive {
        return ErrInvalidStateTransition
    }
    m.Status = MerchantStatusSuspended
    m.addEvent(event.MerchantSuspended{
        MerchantID: string(m.ID),
        OccurredAt: time.Now(),
    })
    return nil
}

// ActiveCredential 查询指定渠道的当前有效凭据（只读辅助方法）
func (m *Merchant) ActiveCredential(channel PaymentChannel) (*ChannelCredential, error) {
    for i, c := range m.Credentials {
        if c.Channel == channel && c.Status == CredentialStatusActive {
            return &m.Credentials[i], nil
        }
    }
    return nil, ErrCredentialNotFound
}

func (m *Merchant) addEvent(e event.DomainEvent) {
    m.Events = append(m.Events, e)
}

// ClearEvents 返回未发布事件并清空，由 UseCase 调用后发布
func (m *Merchant) ClearEvents() []event.DomainEvent {
    events := m.Events
    m.Events = nil
    return events
}
```

```go
package port

import (
    "context"
    "payment-demo/internal/payment/domain/model"
)

// MerchantQuery payment 上下文对 merchant 上下文的 ACL 查询端口
// 消费方（payment）定义，adapter/merchant/ 实现
// ChargeUseCase 通过此端口获取商户渠道凭据，决定路由哪个 Gateway 实例
type MerchantQuery interface {
    // FindActiveCredential 查询指定商户、指定渠道的有效凭据
    // 返回 ErrMerchantCredentialNotFound 若凭据不存在或已吊销
    FindActiveCredential(ctx context.Context, merchantID string, channel model.PaymentMethod) (*ChannelCredentialView, error)
}

// ChannelCredentialView payment 视角的渠道凭据视图
// 只暴露 Gateway 初始化所需的最小字段，隐藏 merchant 侧的实体细节
type ChannelCredentialView struct {
    CredentialID string
    MerchantID   string
    Channel      string            // "CARD" / "PAYPAL"
    Secrets      map[string]string // 传递给 Gateway 工厂用于初始化 HTTP 客户端
}
```

```go
package event

import (
    "time"

)

// DomainEvent 别名，保持与 payment 上下文完全相同的模式
type DomainEvent = sharedEvent.DomainEvent
```

## 设计决策说明

以下是每个区间编排逻辑的关键判断，方便人工确认需求理解是否准确：

### Domain 层（AC-1 ~ AC-19）

| 决策点 | 验收标准体现 |
|--------|------------|
| 不变式：**同一渠道最多一个 ACTIVE 凭据** | AC-4（重复添加拒绝）、AC-6（REVOKED 后可再添加）、AC-7（不同渠道可共存） |
| `Suspend` **仅 ACTIVE → SUSPENDED** 合法，DEACTIVATED 视为非法 | AC-12、AC-13 |
| `RevokeCredential` 需区分**凭据不存在**和**状态非法** | AC-9（已 REVOKED → ErrInvalidStateTransition）、AC-10（不存在 → ErrCredentialNotFound）|
| `AddCredential` 对非 ACTIVE 商户**任何渠道**均拒绝 | AC-5 |
| `ClearEvents` 幂等：连续调用第二次返回空 | AC-16 |

### Application 层（AC-20 ~ AC-29）

| 决策点 | 验收标准体现 |
|--------|------------|
| 外部依赖失败（Repository.Save）**不持久化** | AC-21、AC-23、AC-24、AC-28 |
| `ChargeUseCase` 新增 `MerchantID` 入参，**先查凭据再构造 Gateway** | AC-27、AC-29 |
| MerchantQuery 失败时 **Transaction.Save 不被调用** | AC-28 |

### Handler 层（AC-30 ~ AC-39）

| HTTP 状态码 | 触发场景 |
|------------|---------|
| 201 | 注册商户（AC-30）、添加凭据（AC-33） |
| 200 | 吊销凭据（AC-36） |
| 401 | 未认证（AC-32） |
| 404 | 商户不存在（AC-34）、凭据不存在（AC-37） |
| 409 | 同渠道已有 ACTIVE 凭据（AC-35） |
| 422 | 参数缺失（AC-31、AC-38）、MerchantQuery 找不到凭据（AC-39） |

### Adapter 层（AC-40 ~ AC-49）

| 决策点 | 验收标准体现 |
|--------|------------|
| `Save` 具有 **upsert** 语义（与现有 `TransactionRepository` 保持一致） | AC-42 |
| `sync.RWMutex` 保护**并发写**和**读写混合** | AC-44（纯写并发）、AC-45（读写混合） |
| ACL 适配器**翻译层**：merchant 模型 → payment `ChannelCredentialView` | AC-46 |
| ACL 适配器需区分**商户不存在**和**凭据不存在/已吊销** | AC-47、AC-48 |
| `GatewayFactory` **按 Secrets 动态构造**，不同商户得到不同实例 | AC-49 |
